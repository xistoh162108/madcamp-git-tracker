몰입캠프 GitHub 활동 리더보드 웹사이트의 한국어 UI/UX 디자인을 만들어줘.

이 서비스는 MadCamp 참가자들의 GitHub repository 활동을 기반으로 개인, 팀, 분반별 commit 활동을 시각화하는 인터랙티브 리더보드다. 목적은 참가자를 감시하거나 평가하는 것이 아니라, 캠프 분위기를 게임처럼 재미있게 만들고, 각 팀과 분반의 개발 흐름을 한눈에 보여주는 것이다.

반드시 한국어 UI로 디자인해줘.

전체 톤은 “개발자 캠프 + 게임형 리더보드 + 깔끔한 운영 대시보드” 느낌이면 좋겠다. 너무 기업용 관리자 페이지처럼 딱딱하지 않고, 그렇다고 너무 유치한 게임 UI도 아니게 해줘. KAIST 몰입캠프 분위기에 맞게 세련되고 젊은 느낌이면 좋다.

서비스 이름

임시 서비스 이름은 다음 중 하나로 사용해줘.
- 몰입 랭킹
- MadCamp Leaderboard
- 몰입캠프 GitHub 리더보드

메인 문구 예시:
- “이번 주 가장 뜨겁게 몰입한 팀은?”
- “커밋으로 보는 몰입의 흐름”
- “팀과 분반의 개발 에너지를 한눈에”
- “GitHub 활동으로 보는 이번 주 몰입도”

중요한 서비스 설계 조건

이 서비스는 최대한 서버리스 환경에서 운영하고 싶다.

따라서 UI/UX도 다음 구조를 고려해서 디자인해줘.
- 프론트엔드는 Vercel, Netlify, Cloudflare Pages 같은 서버리스 배포 환경을 가정한다.
- 데이터는 GitHub API에서 주기적으로 수집된다.
- 수집 작업은 cron job, scheduled function, GitHub Actions, Vercel Cron, Cloudflare Workers Cron 같은 방식으로 실행될 수 있다.
- 집계된 데이터는 Supabase, Firebase, Turso, Neon, Cloudflare D1 같은 관리형 DB 또는 서버리스 DB에 저장된다고 가정한다.
- 프론트엔드는 저장된 집계 데이터를 읽어서 리더보드를 보여준다.
- 실시간성이 꼭 완벽할 필요는 없고, “마지막 동기화 시간”을 보여주는 방식이면 충분하다.

즉, 24시간 켜져 있는 전용 서버를 전제로 하지 말고, 서버리스/정적 프론트엔드/주기적 동기화 구조에 어울리는 UI로 설계해줘.

관리자 설정 가능해야 하는 항목

주차 정보는 코드에 고정하지 않는다. 관리자가 설정파일 또는 관리자 페이지에서 바꿀 수 있어야 한다.

관리자가 설정할 수 있어야 하는 값:
- 기수 이름: 예: 2026 여름학기
- 캠프 시작일
- 캠프 종료일
- 현재 활성 주차
- 전체 주차 목록
- 각 주차의 시작일과 종료일
- 분반 목록
- repository naming rule
- GitHub organization 이름
- 집계 대상 repository prefix
- 랭킹 공개 여부
- 특정 repository 제외 목록
- 특정 참가자 제외 목록
- 관리자 계정 목록
- 분반 랭킹 기본 기준: 인당 평균 / 총합
- 리더보드 업데이트 주기
- 마지막 동기화 시간 표시 여부

제공해야 하는 랭킹

이 사이트는 다음 랭킹을 제공한다.
1. 주차별 개인 커밋 수 랭킹
2. 주차별 팀 커밋 수 랭킹
3. 주차별 분반 랭킹
4. 전체 기간 개인 커밋 수 랭킹
5. 전체 기간 팀 커밋 수 랭킹
6. 전체 기간 분반 랭킹

분반 랭킹은 분반별 인원 수가 1~3명 정도 차이 날 수 있다.
따라서 기본 정렬은 “인당 평균 커밋 수”로 보여주고, 토글을 통해 “총 커밋 수”도 볼 수 있게 해줘.

분반 랭킹 UI에는 반드시 다음 토글이 있어야 한다.
- 인당 평균 기준
- 총합 기준

인당 평균 기준일 때는 “분반별 인원 차이를 보정한 기준입니다”라는 작은 설명을 넣어줘.

중요한 표현 정책

이 UI는 참가자를 과도하게 감시하는 느낌이면 안 된다.

피해야 할 단어:
- 감시
- 평가
- 벌점
- 저조
- 부진
- 꼴찌
- 미활동자 압박
- 기여 부족

권장하는 단어:
- 활동
- 몰입
- 흐름
- 기여
- 기록
- 개발 에너지
- 이번 주 움직임
- 팀의 발자국
- GitHub 활동

작은 안내 문구를 반드시 넣어줘.

문구 예시:

“커밋 수는 활동량을 보여주는 참고 지표이며, 프로젝트 기여도 전체를 의미하지는 않습니다.”

또는:

“작은 커밋, 문서 정리, 기획, 디자인, 디버깅도 모두 중요한 기여입니다.”

전체 페이지 구조
1. 메인 대시보드

상단에는 Hero 영역을 둔다.

제목:
“몰입캠프 GitHub 리더보드”

부제:
“팀별 repository 활동을 기반으로 이번 주와 전체 기간의 개발 흐름을 확인하세요.”

상단 요약 카드:
- 총 커밋 수
- 이번 주 커밋 수
- 가장 활발한 팀
- 가장 활발한 분반
- 참여 중인 repository 수
- 마지막 업데이트 시간
- 현재 활성 주차

각 카드는 숫자가 크게 보이게 해줘.
숫자가 부드럽게 올라가는 count-up 애니메이션 느낌이면 좋다.
1. 설정 기반 주차 선택 영역

주차 선택은 하드코딩된 1~4주차 버튼이 아니라, 관리자 설정에서 불러온 주차 목록을 기반으로 보여주는 느낌이어야 한다.

예시 UI:
- 전체 기간
- 현재 주차
- 1주차: 07.02 ~ 07.09
- 2주차: 07.09 ~ 07.16
- 3주차: 07.16 ~ 07.23
- 4주차: 07.23 ~ 08.02

현재 활성 주차에는 “진행 중” badge를 붙여줘.

예시:

“2주차 · 진행 중”
1. 필터 영역

사용자가 랭킹 기준을 바꿀 수 있어야 한다.

필터:
- 기간 선택: 전체 기간 / 현재 주차 / 특정 주차
- 분반 선택: 전체 / 1분반 / 2분반 / 3분반 / 4분반
- 랭킹 유형: 개인 / 팀 / 분반
- 분반 랭킹 기준: 인당 평균 / 총합
- 정렬 기준: 커밋 수 / 최근 활동 / 순위 상승

필터는 상단에 sticky하게 두거나, 대시보드 카드 위에 자연스럽게 배치해줘.
1. 메인 리더보드

리더보드는 단순한 표가 아니라 게임 리더보드처럼 보여줘.

각 row에는 다음 정보가 들어간다.

개인 랭킹:
- 순위
- 참가자 이름
- GitHub username
- 분반
- 현재 주차 팀
- 커밋 수
- 전주 대비 변화량
- 순위 상승/하락 표시
- GitHub profile 링크

팀 랭킹:
- 순위
- 팀 repo 이름
- 분반
- 팀원 목록
- 총 커밋 수
- 인당 평균 커밋 수
- 최근 활동 시간
- 순위 상승/하락 표시
- GitHub repository 링크

분반 랭킹:
- 순위
- 분반
- 총 커밋 수
- 인당 평균 커밋 수
- 참가자 수
- 활성 repository 수
- 전주 대비 변화량

1위, 2위, 3위는 시각적으로 특별하게 보여줘.
메달, 트로피, 빛나는 효과 같은 요소를 사용해도 좋다.
단, 과하게 자극적이지 않게 해줘.
1. Top 3 Podium

리더보드 위나 옆에 Top 3를 podium 형태로 보여줘.

구성:
- 1위는 가운데 크게
- 2위는 왼쪽
- 3위는 오른쪽

개인 / 팀 / 분반 랭킹 유형에 따라 podium 내용이 바뀌는 느낌으로 디자인해줘.
1. 활동 추이 그래프

아래에는 활동 추이 그래프를 넣어줘.

그래프 종류:
- 주차별 총 커밋 수 변화
- 분반별 커밋 수 비교
- 팀별 커밋 수 변화
- 개인별 누적 커밋 수 변화
- 날짜별 전체 커밋 수 변화

그래프는 너무 복잡하지 않게, 운영진과 참가자가 빠르게 이해할 수 있게 해줘.
1. 최근 활동 Feed

오른쪽 사이드나 하단에 최근 활동 피드를 넣어줘.

예시 문구:
- “김가온님이 2026-summer-w2-c1-03에 commit 3개를 추가했습니다.”
- “2분반의 이번 주 총 커밋 수가 500개를 돌파했습니다.”
- “w2-c3-07 팀이 팀 랭킹 3위로 상승했습니다.”
- “마지막 동기화 이후 128개의 새 커밋이 반영되었습니다.”

단, 실제 GitHub commit message를 그대로 크게 노출하지는 말아줘.
민감하거나 부적절한 commit message가 있을 수 있기 때문이다.
활동 feed는 요약형으로 보여줘.
1. Achievement / Badge 요소

게임 느낌을 위해 badge 요소를 넣어줘.

예시 badge:
- 첫 커밋
- 이번 주 10 commits
- 꾸준한 기여자
- 밤샘 개발자
- 문서 정리왕
- 리팩토링 장인
- 마감 전력질주
- README 정리 완료
- 첫 배포 성공
- 연속 활동

다만 badge들은 재미 요소이며 평가 지표가 아니라는 느낌으로 디자인해줘.
1. 서버리스 동기화 상태 UI

서버리스 환경에서는 실시간 업데이트가 아니라 주기적 동기화가 중요하다.

따라서 화면 어딘가에 다음 정보를 보여줘.
- 마지막 동기화 시간
- 다음 예상 동기화 시간
- 동기화 상태: 정상 / 지연 / 실패
- GitHub API rate limit 상태
- 집계 대상 repository 수
- 최근 동기화에서 반영된 commit 수

예시 UI 문구:
- “마지막 업데이트: 2026.07.12 18:30”
- “GitHub 활동은 1시간마다 자동 반영됩니다.”
- “현재 데이터는 마지막 동기화 시점 기준입니다.”
1. 관리자 설정 화면

관리자 전용 화면도 디자인해줘.

관리자 설정 화면에는 다음 섹션이 필요하다.

캠프 설정
- 기수 이름
- season id
- 캠프 시작일
- 캠프 종료일
- 현재 활성 주차
- 분반 목록

주차 설정

주차 목록을 카드 또는 테이블로 보여줘.

각 주차 항목:
- 주차 번호
- 표시 이름
- 시작일
- 종료일
- 상태: 예정 / 진행 중 / 종료
- 수정 버튼

GitHub 설정
- GitHub organization 이름
- repository prefix
- repository naming pattern
- 집계 대상 repository 목록
- 제외 repository 목록
- 제외 참가자 목록

랭킹 설정
- 분반 랭킹 기본 기준: 인당 평균 / 총합
- 개인 랭킹 공개 여부
- 팀 랭킹 공개 여부
- 분반 랭킹 공개 여부
- badge 기능 사용 여부

동기화 설정
- 업데이트 주기
- 수동 동기화 버튼
- 마지막 동기화 시간
- 동기화 로그
- 실패 repository 목록
- GitHub API rate limit 상태

관리자 화면은 참가자용 화면보다 조금 더 실용적인 대시보드 느낌이면 좋다.
1. 특정 팀 상세 화면

팀 상세 화면에는 다음 정보가 들어간다.
- repo 이름
- 주차
- 분반
- 팀 번호
- 팀원 목록
- 팀 총 커밋 수
- 팀원별 커밋 수
- 날짜별 커밋 추이
- 최근 활동
- README 링크
- GitHub repository 링크
- 마지막 activity 시간
1. 특정 참가자 상세 화면

참가자 상세 화면에는 다음 정보가 들어간다.
- 이름
- GitHub username
- 소속 분반
- 주차별 팀 이력
- 주차별 커밋 수
- 전체 커밋 수
- 최근 활동 repository
- 획득 badge
- GitHub profile 링크

주차별 팀 이력 예시:
- 1주차: 2026-summer-w1-c1-01
- 2주차: 2026-summer-w2-c1-03
- 3주차: 2026-summer-w3-c1-05
- 4주차: 예정
1. Empty / Loading / Error 상태

실제 서비스처럼 다음 상태도 디자인해줘.

Loading:
- “GitHub 활동을 불러오는 중입니다…”
- skeleton card 사용

Empty:
- “아직 표시할 커밋 기록이 없습니다.”
- “팀 repository가 생성되면 이곳에 활동이 표시됩니다.”

Error:
- “GitHub 데이터를 불러오지 못했습니다.”
- “마지막 동기화 데이터를 기준으로 표시합니다.”
- “잠시 후 다시 시도해 주세요.”

Serverless sync delay:
- “아직 최신 GitHub 활동이 반영되지 않았을 수 있습니다.”
- “데이터는 주기적으로 자동 업데이트됩니다.”

디자인 스타일

스타일 방향:

최종 산출물

최종적으로 바로 개발에 참고할 수 있는 고품질 UI mockup을 만들어줘.

특히 다음이 잘 드러나야 한다.
- 게임형 리더보드 느낌
- 한국어 UI
- 주차를 관리자 설정으로 바꿀 수 있는 구조
- 서버리스 환경에 맞는 동기화 상태 표현
- 개인 / 팀 / 분반 랭킹 전환
- 분반 랭킹에서 인당 평균과 총합 토글
- 과도한 감시가 아니라 긍정적인 활동 시각화라는 느낌
- 데스크톱과 모바일 대응

추가 요구사항:

주차 설정은 날짜 단위만이 아니라, KST 기준 시작 시각과 종료 시각까지 지정할 수 있어야 한다.

몰입캠프 운영은 한국 시간 기준으로 진행되므로, 모든 주차 계산과 랭킹 집계 기준 시간대는 기본적으로 Asia/Seoul, 즉 KST로 처리한다.

관리자는 설정 화면 또는 설정 파일에서 다음 값을 지정할 수 있어야 한다.
- 전체 서비스 기본 시간대: 기본값 Asia/Seoul
- 각 주차의 시작 날짜와 시작 시각
- 각 주차의 종료 날짜와 종료 시각
- 현재 활성 주차
- 주차별 표시 이름
- 주차별 상태: 예정 / 진행 중 / 종료
- 주차별 집계 포함 여부

예를 들어 2주차가 2026년 7월 9일 오전 9시에 시작해서 2026년 7월 16일 오전 8시 59분 59초에 끝난다면, UI와 집계 기준 모두 KST 기준으로 처리되어야 한다.

관리자 설정 화면의 “주차 설정” 섹션에는 다음 UI가 필요하다.

각 주차를 카드 또는 테이블 row로 보여준다.

각 row에 표시할 값:
- 주차 번호
- 표시 이름
- 시작 일시
- 종료 일시
- 시간대: KST / Asia/Seoul
- 상태: 예정 / 진행 중 / 종료
- 집계 포함 여부
- 수정 버튼

수정 UI에는 다음 입력 필드가 있어야 한다.
- 주차 번호
- 표시 이름
- 시작 날짜
- 시작 시간
- 종료 날짜
- 종료 시간
- 시간대
- 집계 포함 여부

사용자에게 보이는 주차 선택 UI도 날짜뿐 아니라 시간까지 보여줄 수 있어야 한다.

예시:
- 1주차 · 07.02 09:00 ~ 07.09 08:59
- 2주차 · 07.09 09:00 ~ 07.16 08:59 · 진행 중
- 3주차 · 07.16 09:00 ~ 07.23 08:59
- 4주차 · 07.23 09:00 ~ 08.02 23:59

상단 대시보드에는 현재 활성 주차와 해당 주차의 종료까지 남은 시간을 보여줘도 좋다.

예시:
- 현재 주차: 2주차
- 집계 기준: KST
- 2주차 종료까지 3일 04시간 남음
- 마지막 업데이트: 2026.07.12 18:30 KST

동기화 상태 UI에도 시간대가 명확히 보이면 좋다.

예시:
- 마지막 동기화: 2026.07.12 18:30 KST
- 다음 동기화 예정: 2026.07.12 19:30 KST
- 모든 랭킹은 KST 기준 주차 설정에 따라 집계됩니다.

주의할 점:

GitHub API에서 가져오는 commit timestamp는 UTC 또는 작성자 local time과 관련될 수 있으므로, 집계 시에는 모든 시간을 서버 내부에서 KST 기준 구간으로 변환해서 계산하는 것을 전제로 디자인해줘.

UI 안내 문구 예시:

“랭킹은 운영진이 설정한 KST 기준 주차 구간에 따라 집계됩니다.”

“GitHub 활동 시간은 KST로 변환되어 표시됩니다.”

“주차 시작/종료 시각은 관리자 설정에서 변경할 수 있습니다.”

추가 디자인 지시:

Template은 “게임형 리더보드 + 운영 대시보드 + GitHub analytics” 조합으로 잡아줘.

일반적인 SaaS dashboard처럼 너무 딱딱하지 않게 하고, e-sports ranking 화면처럼 순위 경쟁의 재미가 느껴지게 해줘. 하지만 참가자를 감시하거나 평가하는 느낌은 피하고, “몰입의 흐름을 함께 보는 축제형 대시보드”처럼 보여줘.

Design System은 다음 기준으로 설정해줘.
- 전체 언어: 한국어
- 기본 테마: 다크 모드
- 분위기: GitHub + e-sports + KAIST 개발 캠프
- 배경: 검정에 가까운 navy / charcoal 계열
- 주요 색상: electric blue 또는 cyan
- 보조 색상: violet
- 긍정적 변화 색상: lime
- Top 1 강조 색상: gold
- 위험/오류 색상: red를 약하게 사용
- 폰트: 한국어 가독성이 좋은 sans-serif
- 카드: 둥글지만 너무 과하지 않게
- 그림자: subtle glow 중심
- 표: 일반 표가 아니라 leaderboard row 카드처럼 표현
- badge: 게임 achievement 느낌
- 차트: 복잡하지 않고 직관적인 line/bar chart
- 애니메이션: 숫자 count-up, 순위 상승, 동기화 상태 정도만 부드럽게 사용

피해야 할 디자인:
- 너무 흔한 beige/cream/orange Claude 기본 스타일
- 과도한 유리 효과
- 너무 많은 gradient
- 참가자를 압박하는 빨간 경고 UI
- “꼴찌”, “부진”, “감시”, “평가” 같은 느낌
- 기업용 관리자 페이지처럼 딱딱한 표 중심 UI

랭킹 지표 표현은 다음 원칙을 따라줘.
- 공개 리더보드의 기본 지표는 커밋 수로 둔다.
- 커밋 수는 “활동량 참고 지표”로 표현한다.
- Line 수는 메인 랭킹 기준이 아니라 상세 분석 지표로만 표현한다.
- 팀/분반 랭킹에는 인당 평균을 함께 보여준다.
- 분반 랭킹은 기본적으로 인당 평균 기준으로 정렬한다.
- Line 수는 “변경 line 수”로 표시하되, 자동 생성 파일과 대용량 파일은 제외된 참고 지표라는 안내를 넣는다.

UI 안내 문구를 포함해줘.

“커밋 수는 GitHub 활동량을 보여주는 참고 지표입니다.”

“작은 커밋, 문서 정리, 기획, 디자인, 디버깅도 모두 중요한 기여입니다.”

“분반 랭킹은 인원 차이를 보정하기 위해 기본적으로 인당 평균 기준으로 표시됩니다.”

“변경 line 수는 자동 생성 파일과 대용량 파일을 제외한 참고 지표입니다.”
